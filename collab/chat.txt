## What we know about the render pipeline
- `W3DDisplay::draw()` calls `drawViews()` for in-game frames (m_loadScreenRender == FALSE)
- `drawViews()` iterates `m_viewList` and calls `v->drawView()` on each view
- `W3DView::drawView()` is the actual 3D scene renderer (file: W3DView.cpp, line ~1527)
- The view list is populated — TheTacticalView exists (HUD references it)
- ChipsetType = 12 (DC_GENERIC_PIXEL_SHADER_1_4) — pixel shader terrain path is active
- All terrain VBs created and locked successfully (confirmed in startup trace)
- Terrain textures created via D3DX fallback path (fmt=26 = A8R8G8B8 — many fallbacks in trace)
- No device reset between VB creation and in-game rendering
- W3DShroud::ReAcquireResources: dstTex=0B0571B4 (non-NULL, POOL_MANAGED)

## Most likely cause
`W3DView::drawView()` is being called but the 3D terrain draw calls are NOT producing visible output. Possible reasons:
1. **Camera is wrong** — camera position/target outside the map, or view frustum culls terrain
2. **Render object not in scene** — TheTerrainRenderObject might not be added to the W3D scene graph for the in-game map (added for shell map but reset on new map load without re-adding)
3. **Vertex buffer lost** — VBs created but POOL_DEFAULT VBs were released by a device reset that happened AFTER our trace ends, OR VBs created in wrong pool that gets wiped
4. **Z-test failure** — all terrain pixels fail depth test due to wrong near/far clip or inverted depth
5. **Shader error** — chipset 12 pixel shader path produces no output (shader missing or failing to compile/link)
